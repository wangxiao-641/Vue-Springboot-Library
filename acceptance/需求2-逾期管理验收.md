# 需求变化 2：逾期管理和逾期限制规则——验收记录

验收日期：2026-07-16

## 改动范围

- 借书时间和应还时间完全由后端生成，业务时区固定为 `Asia/Shanghai`，应还时间为借书时间后 30 个日历日。
- 后端统一计算 `NORMAL` / `DUE_SOON` / `OVERDUE`，同时返回中文文案和逾期天数。以上海自然日为边界：昨天为 1 天，今天到期不逾期，0–3 天为即将到期。
- `GET /bookwithuser` 和 `GET /LendRecord` 支持 `overdueOnly=true`；当前借阅和未还历史记录均附带应还日期、稳定状态和逾期天数。
- 新增管理员专用 `PUT /bookwithuser/due-date`，只接收 `operatorId` / `borrowId` / `dueDate`，仅更新 `deadtime`。非管理员被拒绝，旧当前借阅任意新增/更新接口也已拒绝直写。
- `CirculationService.borrowBook` 在任何库存或记录写入前拒绝有逾期未还图书的读者；`renewBook` 在更新前拒绝逾期目标。归还后删除当前借阅，限制自动解除。
- 当前借阅页显示应还日期、状态、逾期天数和明确逾期提示；管理员可筛选逾期未还并在专用弹窗中调整应还日期。借阅记录页同步提供逾期筛选和到期信息。
- 新增 `verify_overdue_http.py` 纯 HTTP 黑盒脚本、已有数据库应还日期完整性迁移、基线 SQL、运行指南和项目地图更新。
- 基线 `bookwithuser.deadtime` 改为 `NOT NULL`。逾期迁移在任何 ALTER 前检查 NULL，发现问题用 `SIGNAL 45000` 明确中止；数据完整时幂等改为 `NOT NULL` 并添加索引。

## 残余风险

- 项目仍没有统一后端 token 鉴权拦截器。专用日期调整接口会校验 `operatorId` 确实对应管理员，但无法证明请求发起者持有该管理员的 token；这是项目已知架构缺口，本需求未扩展为整体鉴权改造。
- 旧 `/LendRecord/*` 编辑/删除和 `/bookwithuser/delete*` 仍可绕过 `CirculationService`。本需求已关闭当前借阅的任意新增/更新，但未删除旧管理功能和黑盒清理所依赖的删除端点。
- `LendRecord` 逾期筛选依据当前借阅表的应还日期，因为现有历史表没有独立 `deadtime` 字段；已归还历史不会被“逾期未还”筛选命中，符合本次需求。
