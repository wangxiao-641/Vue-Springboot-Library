# 需求变化 4：管理员新增和删除用户——验收记录

验收日期：2026-07-17

## 改动范围

- 用户管理页新增明确的“新增读者”入口，表单包含用户名、初始密码、姓名、电话、性别、地址，并提供必填与格式提示。新增角色在前后端均固定为 `role=2`。
- `POST /user` 改为管理员新增普通读者专用接口，严格校验用户名、密码、姓名及可选基本字段，并校验 `operatorId` 对应管理员。
- `user.username` 新增唯一索引 `uk_user_username`，并提供幂等迁移。迁移在任何 ALTER 前检查旧库重复用户名，发现重复即 `SIGNAL 45000` fail-fast。
- `DELETE /user/{id}?operatorId=` 只允许管理员删除普通读者；管理员目标、不存在的读者、有 `bookwithuser` 当前借阅的读者均明确拒绝。
- 删除与借书都先对读者账号行加锁，避免“删除检查无借阅后又并发借出”的竞态。
- 前端删除前有二次确认；新增或删除失败时仅显示后端原因，不关闭新增弹窗、不显示成功。
- 前端已移除用户批量选择/批量删除入口；旧 `POST /user/deleteBatch` 后端路径保留兼容，但一律明确拒绝。
- 新增 `verify_user_management_http.py` 纯 HTTP 黑盒脚本，并更新基线 SQL、迁移、`run_guide.md`、项目地图和最终 `vue/dist`。

## 残余风险

- 项目仍无统一后端 token 鉴权拦截器。本需求的新增/删除接口会校验 `operatorId` 确实对应 `role=1`，能拒绝普通读者 ID，但无法证明请求发起者持有该管理员的 token/会话。
- 既有 `PUT /user`、`PUT /user/password`、公开注册和读者列表查询的整体权限模型仍是项目遗留问题；本次按要求没有扩展为复杂鉴权系统。
- 密码仍沿用项目原有的明文存储/比对，本次为确保新账号能用初始密码登录而保持兼容，未扩展密码体系重构。
